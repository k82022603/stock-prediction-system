# AI 시대의 소프트웨어 개발 방법론 - Part 2
## 실전 AI 개발: 프로젝트 라이프사이클

### 서론: 이론에서 실전으로

Part 1에서 우리는 전통적 개발 방법론에서 AI-Native 개발로의 패러다임 전환을 살펴보았습니다. Part 2에서는 실제 프로젝트에서 AI를 어떻게 활용하는지, 구체적인 단계와 기법들을 다룹니다.

이 문서는 React + Spring Boot + MyBatis + PostgreSQL 기반 주식 예측 시스템을 4-5시간 만에 구축한 실제 경험을 바탕으로 작성되었습니다.

## 제1부: 프로젝트 시작 - AI와의 첫 대화

### 1.1 초기 프롬프트의 중요성

전통적 개발에서 "요구사항 정의서"가 프로젝트의 청사진이었다면, AI 개발에서는 "초기 프롬프트"가 그 역할을 합니다.

#### 좋지 않은 프롬프트 예시:

```
주식 예측 시스템 만들어줘
```

**문제점:**
- 기술 스택 불명확
- 시스템 범위 불명확
- 아키텍처 스타일 불명확

#### 개선된 프롬프트 예시:

```
React 18 + Spring Boot 3.2 + MyBatis 3.0 + PostgreSQL 15 기반의
내일 한국 주식 시장 예측 시스템을 만들어줘.

요구사항:
1. 백엔드: RESTful API, KOSPI/KOSDAQ 종목 관리
2. 프론트엔드: 반응형 UI, 차트 시각화
3. 데이터베이스: 종목 정보, 예측 결과, 사용자 관리
4. 배포: Docker 기반, 개발/운영 환경 분리
```

### 1.2 생성된 코드 검토하기

AI가 생성한 코드를 맹목적으로 신뢰해서는 안 됩니다. 반드시 검토해야 합니다.

#### 체크리스트:

**1) 아키텍처 검토**
- 레이어 분리가 적절한가?
- 의존성 주입이 올바른가?

**2) 보안 검토**
- SQL Injection 방어
- XSS 방어
- 인증/인가 처리

**3) 성능 검토**
- N+1 쿼리 문제
- 인덱스 정의
- 캐싱 필요성

## 제2부: 반복적 개선 - AI와의 협업

### 2.1 피드백 루프

전통적 개발: 구현 → 테스트 → 버그 수정 (일 단위)
AI 개발: 구현 → 검토 → 개선 (분 단위)

### 2.2 실제 개선 사례들

#### 사례 1: API 응답 형식 표준화

**개선 요청:**
```
모든 API 응답을 표준 형식으로 통일해줘.
success, data, message, timestamp 포함.
```

#### 사례 2: 프론트엔드 에러 처리 개선

**개선 요청:**
```
fetch 호출에 에러 처리, 로딩 상태, 타임아웃 추가해줘.
```

## 제3부: 테스트와 품질 보증

### 3.1 AI와 함께하는 테스트 작성

**프롬프트:**
```
StockService에 대한 JUnit 테스트 작성해줘.
Mockito 사용, 정상 케이스와 예외 케이스 커버.
```

### 3.2 테스트 커버리지 확보

**프롬프트:**
```
테스트 커버리지 80% 이상으로 만들고 싶어.
Service와 Controller 레이어 모든 메서드 테스트.
```

## 제4부: 배포와 운영

### 4.1 Docker 컨테이너화

**프롬프트:**
```
이 프로젝트를 Docker로 컨테이너화해줘.
docker-compose.yml과 각 Dockerfile 작성.
```

### 4.2 CI/CD 파이프라인

**프롬프트:**
```
GitHub Actions로 CI/CD 파이프라인 구성해줘.
PR 시: 빌드, 테스트, 코드 분석
main push 시: Docker 이미지 빌드 후 배포
```

## 제5부: 모니터링과 유지보수

### 5.1 로깅 전략

**프롬프트:**
```
프로덕션 레벨의 로깅 시스템 구축해줘.
환경별 로그 레벨, 파일 rotation, JSON 형식.
```

### 5.2 성능 모니터링

**프롬프트:**
```
Spring Boot Actuator로 헬스체크와 메트릭 수집.
Prometheus 형식으로 export.
```
